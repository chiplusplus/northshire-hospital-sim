# Northshire Hospital Simulator

Simulated NHS Trust environment for the [Access-IQ Platform](https://github.com/chiplusplus/access-iq-platform). This repo generates synthetic but structurally realistic healthcare data and deploys it to AWS services that mimic a real Trust's data estate.

**This is not the main project.** It exists to provide the upstream data sources that Access-IQ ingests. See the [Access-IQ Platform](https://github.com/chiplusplus/access-iq-platform) for the analytics pipeline, data modelling, and dashboard.

## What It Creates

The simulator deploys a CDK stack to a separate AWS account (the "Trust account") containing:

| Service                        | What it holds                                                                | How Access-IQ consumes it               |
| ------------------------------ | ---------------------------------------------------------------------------- | --------------------------------------- |
| **RDS Postgres**               | EHR database (patients, encounters, referrals, diagnoses) + urgent care logs | `ingest-postgres` via VPC peering       |
| **AWS Transfer Family** (SFTP) | Daily appointment file drops                                                 | `ingest-sftp` via VPC peering           |
| **S3 bucket**                  | Diagnostics orders (CSV) + provider/site reference (Excel)                   | `ingest-trust-s3` via cross-account IAM |

Data is generated by Python scripts in `src/northshire_sim/generators/` — ~100K synthetic patients, ~586K encounters, referrals, diagnostics orders, urgent care logs, and provider rosters. All data uses realistic NHS structures (NHS numbers with valid Mod-11 checksums, LSOA codes, specialty codes) but is entirely synthetic.

## Setup

**Prerequisites:** Python 3.12+, AWS CLI v2 with an SSO profile for the Trust account, Node.js (for CDK).

```bash
git clone https://github.com/chiplusplus/northshire-hospital-sim.git
cd northshire-hospital-sim
python -m venv .northshire-hospital-sim
source .northshire-hospital-sim/bin/activate
pip install -r requirements.txt
```

## Usage

This repo is typically driven by the platform repo's `make up` / `make down` commands, which call `make trust-bootstrap` and `cdk destroy` automatically. You only need to run commands here directly for standalone development or debugging.

### Managed by Access-IQ (normal workflow)

From the platform repo:

```bash
export TRUST_PROFILE=<your-trust-profile>
export AWS_PROFILE=<your-platform-profile>
make up    # deploys Trust + Platform, seeds data, starts pipeline
make down  # destroys everything
```

The platform's `session.sh` expects this repo as a sibling directory:

```
parent/
├── access-iq-platform/
└── northshire-hospital-sim/
```

### Standalone commands

| Command                        | Description                                                                  |
| ------------------------------ | ---------------------------------------------------------------------------- |
| `make trust-bootstrap`         | Full deploy: CDK stack + DB setup + data generation + publish to all sources |
| `make trust-bootstrap-no-sftp` | Same, without Transfer Family (saves ~$0.30/hr)                              |
| `make trust-seed`              | Seed data into already-deployed infrastructure                               |
| `make trust-destroy`           | Kill tunnels + destroy all AWS infrastructure                                |
| `make cdk-deploy`              | Deploy/update infrastructure only                                            |
| `make generate`                | Generate synthetic data to `data/staging/`                                   |

### Local development (Docker)

For local development without AWS, Docker Compose provides Postgres instances:

```bash
make up         # Start Postgres containers
make generate   # Generate synthetic data
make trust-fast # Publish to local Docker services
make down       # Stop containers
```

## Repo Structure

```
northshire-hospital-sim/
├── src/northshire_sim/
│   ├── generators/      # Pure dataframe generators (patients, encounters, etc.)
│   ├── exports/         # Build files/feeds from generated dataframes
│   ├── checks/          # Cross-entity consistency validation
│   └── publishing/      # Write to DB, S3, SFTP
├── scripts/             # CLI entrypoints (generate, publish, bootstrap)
├── sql/                 # DDL for EHR and urgent care databases
├── infra/               # CDK app (RDS, Transfer Family, S3, VPC)
├── config/              # sources.yaml (generated from CDK outputs)
└── data/                # Generated artefacts (gitignored)
```
